Course: Database Systems (DBS) — Semester 6

1. Introduction

This report documents the complete cloud hosting and deployment process for the ERP Clone Project — an Educational Institute Management System (EIMS) integrated with a Banking Module. The project comprises four services deployed across three cloud platforms, all connected to a centralized PostgreSQL database.

8. Issues Encountered & Resolutions

#IssueRoot CauseResolution
1Backend could not connect to Supabase databaseRender free tier uses IPv4 only; Supabase direct connection uses IPv6Switched to Supabase Session Pooler URL which supports IPv4
2Vercel frontend build failingCI=true (default) treats ESLint warnings as errorsAdded CI=false as environment variable on Vercel
3GitHub push returning 403 ForbiddenPersonal Access Token lacked "repo" scope permissionRegenerated PAT with correct repository permissions
4GitHub push blocked by file size limit156 MB zip file was being tracked by gitRemoved file from tracking and added to .gitignore
5Student profile update returning 500 errorEmpty strings for integer FK columns (department_id) caused PostgreSQL type cast errorApplied "|| null" coalescing to convert empty strings to NULL
6Transaction queries failing"type" is a PostgreSQL reserved word used as column nameRenamed column to "transaction_type" in schema and all backend code
7Backend failing to start on RenderIncorrect start command (npm app.js instead of node app.js)Corrected start command to "node app.js"

9. Security Measures

  • Password Hashing: All passwords hashed using bcrypt with salt rounds of 10
  • Password Policy: Minimum 8 characters, at least 1 uppercase letter, 1 number, and 1 special character
  • SQL Console Security: Admin role verification, DDL/DCL operations blocked (DROP, ALTER, TRUNCATE, CREATE, GRANT, REVOKE), SELECT results capped at 100 rows, 5-second query timeout, all queries logged to Admin_Query_Logs
  • Input Validation: course_offering_id coerced to integer, empty strings converted to NULL for FK fields
  • Environment Variables: Database credentials and API URLs stored as environment variables, never hardcoded
  • CORS: Cross-origin resource sharing enabled for frontend-backend communication

10. Deployment Workflow

The project uses a fully automated CI/CD pipeline:

Step 1: Edit code locally in VS Code
Step 2: Stage and commit changes: git add -A && git commit -m "description"
Step 3: Push to GitHub: git push origin main
Step 4: Render auto-deploys both backends (~2–5 minutes)
Step 5: Vercel auto-deploys both frontends (~1–2 minutes)
Step 6: Verify at live URLs — no manual redeploy needed

11. Conclusion

The ERP Clone Project was successfully deployed to production using a three-platform architecture — Supabase for the database, Render for backend services, and Vercel for frontend hosting. All four services are live, connected, and auto-deploy from a single GitHub repository. The database contains 36 tables, 21 views, 19 triggers, and 35 functions supporting the complete EIMS and Banking workflows. Seven deployment issues were identified and resolved during the hosting process, and security measures were implemented across all layers of the application.
